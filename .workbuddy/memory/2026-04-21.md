### 性能诊断与修复（用户反馈算法不准、数据不实时）

**根因分析：** 发现 4 个影响算法准确性和实时性的性能问题

**已修复：**
1. **useWorkout.ts** — `setCount()` 每帧无条件调用 → 改为仅在 count 变化时触发（用 prevCountRef 记忆上一次值），消除 ~30 次/秒无意义 React re-render
2. **PoseDetectionService.ts** — `getKeypoint()` 每帧做 O(n) Array.find × 12 次 → 改为预构建 `Map<string, Keypoint>` O(1) 查找，在 `CameraView.onResults` 入口一次性构建
3. **CameraView.tsx** — rAF 循环不等 `pose.send()` 完成就排下一帧 → 加 `processingRef` 锁，推理完成后才发下一帧，防止帧队列堆积导致延迟
4. **CameraView.tsx** — `drawSkeleton` 每帧重设 `canvas.width/height` → 改为仅在尺寸变化时重设，避免每帧 GPU 重分配

**新增诊断：** `pose.onResults` 回调中添加 FPS 计数，每秒输出 `[AI Sport] Pose FPS: N` 到控制台

### 致命 BUG 修复 + WorkoutPage 重构（用户反馈无法识别运动 + 界面拥挤）

**根因：两个致命 BUG 导致点击开始后摄像头无法识别运动姿态**

**BUG 1 — CameraView rAF 循环立即退出**
- `loop()` 第一行 `if (!isActiveRef.current) return` 在 startCamera 异步执行后，isActiveRef 可能尚未更新
- rAF 循环启动后瞬间退出 → MediaPipe 推理从未执行 → 零帧数据
- **修复：** rAF 循环不再依赖 isActive 退出，改由 cameraState === 'ready' 控制是否发送推理请求。isActive 变化只控制循环启停

**BUG 2 — onPoseDetected 闭包绑定 startCamera 导致无限重建**
- `startCamera` 的 deps 包含 `onPoseDetected`，而 `handlePoseDetected` 每次重渲染都变
- 导致 startCamera 反复重建 → useEffect([isActive, startCamera]) 反复执行 → 摄像头/MediaPipe 反复销毁重建
- **修复：** 用 `onPoseDetectedRef` (useRef) 存回调，startCamera 不再依赖任何 props/回调

### 摄像头初始化失败修复（用户反馈 `$ is not a constructor`）

**根因：** `import('@mediapipe/pose')` 动态导入在 Vite 生产构建后失败。`@mediapipe/pose@0.5.1675469404` 是 Closure UMD 格式，用 `G("Pose", od)` 导出全局变量，但 Vite ESM 环境下 `this` 是 `undefined`，导致导出注册失败。

**修复：**
1. `index.html` 添加 `<script src="https://cdn.jsdelivr.net/npm/@mediapipe/pose@0.5.1675469404/pose.js">` 全局加载
2. CameraView 改为 `(window as any).Pose` 获取构造函数，不再用动态 import
3. `startLoop` 闭包改用 `cameraStateRef.current` 读取最新状态
4. `getUserMedia` 三级 fallback（精确 deviceId → 宽松分辨率 → `{ video: true }`）
5. CSP 添加 `'unsafe-eval'` + `mediastream:` + `wss:`
### 跳绳计数算法 V3 重写（精准状态机）

**问题：** V2 用峰值检测，阈值 0.015 太低，骨骼点微动就计数。

**重写为 4 状态机：**
```
STANDING → ASCENDING → AIRBORNE → LANDING → count++ → STANDING
```

**核心改进：**
1. 4 状态机要求完整跳跃周期（站立→上升→滞空→落地稳定）
2. 踝关节离地验证：ankleLift >= bodyHeight × 6%（自适应阈值）
3. 最小滞空 3 帧（~100ms）、最大 25 帧（~830ms）
4. 最小落地稳定 2 帧防止弹跳二跳
5. 两次计数最小间隔 6 帧（~200ms）
6. 动态基线（3s 滑动窗口中值），仅在 STANDING 时更新
7. 卡尔曼滤波保留（processNoise 0.0005, measurementNoise 0.006）
8. 速度变号检测（上升→下降 = 到达最高点）

**参考来源：** chenwr727/RopeSkippingCounter（状态机+迟滞）、CSA 2023 论文（踝离地）、Dylan Plummer（归一化位移）
